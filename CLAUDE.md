## シミュレータが再現すべきゲームルール

以下は本ゲームの中核メカニクスの要約。実装時の状態設計・アクション設計の基礎とする（コミュニティの二次資料からの再構成で、数値の正確性は未保証な部分がある）。

### ゲームの目的
2〜4人。各プレイヤーは王国を運営し、マーケットカードを設置してVP（勝利点）を稼ぐ。誰かが**20VP以上**に到達した時点で、残りのプレイヤーが1回ずつ手番を行いゲーム終了。最終的に最もVPが高いプレイヤーが勝利（同点は設置済みマーケットカード枚数で決着、それでも同点なら引き分け）。

### リソースモデル
- **サイコロ**: 基本3個（キャラクターにより増減あり）。1ターンに振り直しはなく、その目を使って各種アクションを行う。
- **パワー**: 3種類（お金・権力・魔力）。同種パワー2個 → 別種パワー1個への変換が可能。
- **サイコロパンカード**: 準サイコロ的リソース。カード獲得・サイコロ目変更以外の用途ではサイコロと同様に使える。
- **手札上限**: 基本6枚（倉庫カードなどで増加）。

### ターンの状態遷移（実装の基本単位）
1. **フェイズ1（ターン開始時）**: キャラクター能力・設置済みカードの「ターン開始時」効果を、任意の順序で処理。
2. **フェイズ2（ターン中の行動）**: サイコロ3個を振った後、以下のアクションを任意の順序・任意回数実行できる（＝行動の組み合わせ探索がシミュレータのコア）。
   - A. カード獲得（サイコロ1個消費、場or山札から手札へ。同名カードが場に複数あれば全獲得）
   - B. カード設置（手札から左上コスト支払い。設置ターンは自分の効果を使用不可＝「設置中」状態を持つ）
   - C. 設置済みカードの使用（「ターン中1回使用」効果。使用後は横向き＝使用済みフラグ）
   - D. レジェンドカード設置（1ターン1回のみ。設置後、同列のレジェンドⅡを補充）
   - E. オープンマーケット入替（3種パワー1個ずつ消費、場8種を全て入替）
   - F. パワー獲得（サイコロ/サイコロパン1個消費）
   - G. パワー変換（同種2個 → 異種1個）
3. **フェイズ3（ターン終了時）**: 手札を上限まで捨てる、使用パワーをストックへ戻す、サイコロパンは捨て札へ、使用済みマーケットカードは未使用状態に戻す、場札を8種に補充（＝補充はターン終了後というタイミングがルール上重要）。

### カードカテゴリ（本体・拡張共通の型）
- **キャラクターカード**: プレイヤーごとに1枚選択。固有能力を持つ（本体8種＋拡張ごとに数種追加）。
- **スタンダードカード**: 本体10種、毎試合すべて使用。基礎の資源生産・手札拡張・ドロー系。
- **アドバンスカード / マーケットカード**: 本体20種中10種を毎試合使用（拡張は追加のカードプールを持つ）。中〜高コストの効果カード群。
- **レジェンドカードⅠ/Ⅱ**: 高VPの目標カード。1ターン1枚までしか設置できない全体制約が存在。
- **追加ルールカード**（一部拡張のみ）: ゲーム全体のルールを変更するメタカード。

## アーキテクチャ（`shared` パッケージ）

ゲームロジックは `shared/src` に、サーバーからもクライアントからも参照できる純粋なTypeScriptとして実装している。パッケージ名は `@mk-online/shared`。

- `types.ts` — `GameState`（部屋1つ分の全状態）、`PlayerState`、`CardCost` / `MarketCardDefinition` などのマスタ型。カードのコスト・VPが未確認の場合は `cost: null` / `victoryPoints: null` で表す規約。
- `actions.ts` — フェイズ2のA〜G（`ACQUIRE_CARD` 〜 `CONVERT_POWER`）に加え、`DRAFT_PICK` / `RESOLVE_TURN_START` / `END_TURN` / `DISCARD_TO_HAND_LIMIT` を含む `GameAction` 判別共用体。UI・サーバー間のプロトコルもこの型がそのまま使える想定。
- `cards/` — キャラクター・スタンダード・アドバンス・レジェンドのマスタデータ（本体のみ）。**コスト等の数値が未確認のカードは `hasProgrammaticEffect: false` とし、エンジン側で意図的に「設置・使用不可（NotImplementedError）」にしている**（数値を捏造してバグを埋め込まないためのガード）。データの確度は `cards/README.md` 参照。
- `engine/reducer.ts` — 中核。`applyAction(state, action, rng): EngineResult<GameState>` という純粋関数（reducerパターン）で全状態遷移を行う。例外を投げず `{ ok: true, value }` / `{ ok: false, error }` を返す（`engine/errors.ts`）。乱数は呼び出し側から `Rng` を注入する設計（`rng.ts` の `createSeededRng`＝再現可能／テスト用、`createCryptoRng`＝本番用）。
- `engine/setup.ts` — 部屋作成〜ドラフト開始までのセットアップ（標準カードのドラフト用パケット配布、アドバンス10種のランダム選出、レジェンド列3枚ずつの配置など）。
- `engine/draft.ts` — 開始時ドラフト（5枚配布→1枚キープ→左隣へパスを3ラウンド）の状態遷移。
- `engine/market.ts` — オープンマーケット（場札8種類）の補充ロジック。既存の場札は維持したまま種類数8まで「トップアップ」する点に注意（`REFRESH_MARKET` で全入替する場合は事前に空にしてから呼ぶ）。
- `engine/view.ts` — `getPlayerView(state, viewerId)`。自分以外の手札・ドラフトパケット・山札の中身を伏せ札にする。**サーバーは必ずこれを通してから `game:update` を送ること**（手札は非公開情報というルールをサーバー側で担保するため）。
- `engine/testUtils.ts` — テスト用に最小構成の `GameState` を組み立てるヘルパー（本体コードではない）。

### 設計上のポイント

- **サーバー権威**: ダイス乱数・コスト検証・勝敗判定はすべて `applyAction` の中で行い、クライアントの自己申告を信用しない（requirements.md 5章）。`server/src/index.ts` の `game:action` ハンドラも、クライアントが送った `playerId` を使わずソケット認証済みの値に必ず差し替える。
- **未実装カードは動かして良い形にしない**: `cost`/`victoryPoints` が確認できるまで、そのカードに関わる `INSTALL_CARD` / `PLACE_LEGEND_CARD` / `USE_INSTALLED_CARD` は `NotImplementedError` を返す。カードデータを埋めるまでUIから普通に弾かれるのが正しい状態。
- **ダイス目とパワー種別の対応（`DIE_FACE_TO_POWER`）は公式ルールブックPDFで確認済み**（1・2→お金／3・4→権力／5・6→魔力）。`types.ts` のTODOコメントは解消済み。
- レジェンドⅠ/Ⅱの補充ルール（`engine/reducer.ts` の `placeLegendCard`）も、公式ルールブックPDFで裏付けが取れた（レジェンドⅠ設置後、隣接するレジェンドⅡを跡地に配置する3列構成）。

### 実装上の注意（紛らわしい挙動）
- 場札の通常補充は**ターン終了後**に行われる（ターン中に減った場札はそのターン中は埋まらない）。ただしE（オープンマーケット入替）は例外で、入替と同時に即座に8枠まで補充される。
- 「獲得」＝手札に加える行為のみを指す。直接設置（キャラ能力・レジェンド設置等）は獲得にカウントされない。
- カード設置（B）の回数制限は公式ルールブックに記載がない（任意回数・コストが払える限り可能）。~~「1ターンに1種類1枚まで」という制限~~は公式ルールブックに根拠がなく誤りだったため削除（2026-07-11、公式ルールブックPDFで確認）。
- 手札上限（`handLimit`）は**マーケットカード＋サイコロパンカードの合計**に対して適用される。手札枚数オーバー分の破棄は自分のターン終了時に処理する。
- カード獲得（A）に使えるのはサイコロのみ（サイコロパンカードは不可）。設置コスト（B）の支払いはサイコロ・サイコロパン併用可。
- マーケットカードの設置コストのサイコロに「Any」表記がある場合のみ1〜6の任意の出目でよい（Any表記がない場合は特定の出目を要求するコストが存在しうることを示唆。本体カードプールに該当カードが実在するかは未確認）。
- 魔導書（C行動の一例）は自分のターン中・自分が振ったサイコロの出目にのみ使用可。サイコロパンカードの出目や他プレイヤーのサイコロは変更不可。
- 【条件発動】と記載されたカード効果は、他プレイヤーの行動をトリガーに発動する。発動条件を満たした時点でカード名を宣言する必要があり、見逃すと発動できない（デジタル実装ではサーバー側で自動判定・通知する設計が必要）。

## サーバー／クライアント構成

- `server/` — Express + Socket.IO。`roomStore.ts` が部屋（インメモリ、6文字コード）を管理し、`index.ts` がSocket.IOイベント（`room:*` / `lobby:*` / `game:action`）を処理して `shared` のルールエンジンを呼び出す。
- `client/` — React + Vite。`HomeScreen`（部屋作成/参加）→ `LobbyScreen`（キャラ選択・初期パワー配分・開始）→ `GameScreen`（ドラフト／ターン進行）の3画面構成。
- `shared` は `vite.config.ts` の `resolve.alias` でソース（`.ts`）を直接参照している（ビルド成果物を経由しない）。

## カードデータの確度（本体のみ実装済み）

本体マーケットカード（スタンダード10種・アドバンス20種・レジェンドⅠ/Ⅱ計10種・キャラクター8種）の設置コスト・VP・能力テキストは、2026-07-11にコミュニティ攻略記事のカード写真で確認済み。詳細・既知の確度不安・未対応の型的な制約（サイコロの出目条件の未検証、一部レジェンドⅡの捨て札コストが未表現）は `shared/src/cards/README.md` を参照。

`CardCost` には以下を追加済み（すべて `engine/reducer.ts` の `placeLegendCard` で実装・テスト済み）:
- `requiredDiscards`: 特定カードを指定枚数、手札または設置済みから捨て札にする設置コスト（例: レジェンドⅡの多く）
- `requiredDiceBreadCount`: サイコロパンを指定枚数（目は問わない）捨てる設置コスト（例: 黄昏の麦隴）
- `requiredAnyHandDiscardCount`: 手札のマーケットカードを種類を問わず指定枚数捨てる設置コスト（例: ポレポレ鳥の秘境。`PlaceLegendCardAction.anyDiscardCardIds` でどのカードを捨てるかプレイヤーが指定する）
- `requiredPipSum`: このターンに振ったダイス・所持サイコロパンの出目合計が指定値以上になるよう使用する設置コスト（例: 無限鏡の大迷宮＝36。`PlaceLegendCardAction.dieIndices` / `diceBreadIndices` で充てる対象を指定する）

`MarketCardDefinition` には `victoryPointsFormula`（盤面状況に応じて動的に決まるVPの計算式。データのみでエンジン側の計算実装はまだ）を追加済み。

この結果、本体マーケットカード全40種＋キャラクター8種のうち `cost: null` のカードはなくなった（全カードが設置可能な状態）。残る既知の制約は `shared/src/cards/README.md` を参照（ダイスの出目一致検証が未実装、五ツ星のネックレスのパワー内訳の確度、各カード固有の能動効果自体は `hasProgrammaticEffect: false` のまま＝コスト受け払いは動くが効果本体は未実装、など）。

`PlayerState.diceBreadCards` は `number`（枚数のみ）から `readonly DieFace[]`（出目つきの配列）に変更済み。ただしサイコロパンのドロー処理（山札から引く処理）自体がまだ未実装のため、現状は消費（支払い）方向のロジックのみ動作する。
