# 要件定義書：某魔法王国ボードゲーム オンライン対戦シミュレータ

最終更新: 2026-07-10（初版）

> このリポジトリは公開を前提とするため、対象ボードゲームの正式タイトル・発行元名は本文中に記載しない（意図的な伏せ字）。

## 1. 背景・目的

個人の趣味・研究として、某魔法王国ボードゲーム（国産の拡大再生産系ボードゲーム）をWeb上で友人・知人と対戦できるツールを作る。将来的には対戦データを使った研究（戦略分析など）に発展させる余地を残すが、**今回のスコープはあくまで「人間同士がオンラインで実際に対戦できること」**を最優先とする。

## 2. スコープ

### 2.1 対応する範囲（MVP）
- Webブラウザで動作するオンライン対戦アプリ
- 「部屋（ルーム）」を作成し、コード/URLを共有して人間プレイヤーが参加
- 定員は本来2〜4人だが、**AIボットで埋めない。集まった人数（2人以上）でそのままゲームを開始できる**
- 対象ボードゲーム**本体**のルール・カードのみを実装（キャラクター8種、スタンダードカード10種、アドバンスカード20種中10種使用、レジェンドカードⅠ・Ⅱ）
- サーバーがルール（サイコロ、パワー、カード設置条件、勝利判定等）を権威的に管理し、全プレイヤーへゲーム状態をリアルタイム同期

### 2.2 今回は対象外（将来検討）
- AIボット対戦相手（人数が足りない枠を埋める機能）
- 拡張パック群のカード・キャラクター
- アカウント登録・ログイン機能（今回は匿名ニックネームのみ）
- 観戦専用モード、途中参加
- 対局リプレイ・統計分析などの研究機能（本来の目的だが、まずは対戦基盤を作ってから着手）
- モバイルアプリ化・オフライン対応

## 3. 用語定義

| 用語 | 意味 |
|---|---|
| 部屋（ルーム） | 1回の対戦セッションの単位。ホストが作成し、コード/URLで人を招待する |
| ホスト | 部屋を作成したプレイヤー。開始操作を行う権限を持つ |
| ゲームエンジン | サーバー側で稼働する、ルール判定と状態遷移を担う純粋ロジック部分 |
| ゲーム状態（State） | サイコロ目・パワー・手札・場札・VP等、対局の全情報 |

## 4. 機能要件

### 4.1 部屋（ルーム）機能
- ホストが部屋を作成すると、部屋コード（短い英数字）と共有URLが発行される
- 参加者はURLアクセス、またはトップ画面でコード入力＋ニックネーム入力により入室する
- 入室できるのは**ゲーム開始前まで**。開始後の途中参加は不可
- ホストは、**2人以上**集まった時点で任意のタイミングでゲームを開始できる（4人未満でも開始可）
- 部屋には有効期限を設ける（例：一定時間操作がなければ自動破棄）。具体的な時間は実装時に決定

### 4.2 プレイヤー参加・退出
- アカウント登録は不要。ニックネームのみで参加
- ゲーム中の切断・離脱時の挙動は**要検討事項**（4章末の「未確定事項」参照）。MVPでは最低限、「切断してもそのプレイヤーの手番は他プレイヤーを止めない」ような配慮が必要か検討する

### 4.3 ゲーム進行（ルールエンジン）
CLAUDE.md に記載した以下の状態遷移をサーバー側のゲームエンジンとして実装する。

- **フェイズ1（ターン開始時）**：キャラクター能力・設置済みカードの開始時効果を、手番プレイヤーが任意の順で処理
- **フェイズ2（ターン中）**：サイコロを振り（サーバー側で乱数生成、クライアントの自己申告は信用しない）、以下のアクションを任意順・任意回数で実行可能
  - A. カード獲得／B. カード設置／C. 設置済みカードの使用／D. レジェンドカード設置（1ターン1枚まで）／E. オープンマーケット入替／F. パワー獲得／G. パワー変換
- **フェイズ3（ターン終了時）**：手札上限までの破棄、パワー精算、場札補充（**ターン終了後に補充**というタイミングを厳守）
- 終了条件：誰かが20VP以上に到達 → 残りのプレイヤーが1巡 → 最終VP比較（同点は設置済みマーケットカード枚数で決着）
- 公式FAQに記載された細かい裁定（本体分・CLAUDE.md参照）をルールエンジンの単体テストとして担保する

### 4.4 UI/UX要件
- ロビー画面：部屋作成／参加、ニックネーム入力、参加者一覧、開始ボタン（ホストのみ）
- ゲーム画面：自分の手札、自分のパワー、オープンマーケット（場札）、レジェンドカード列、他プレイヤーの公開情報（設置済みカード・VP・パワー概数）、サイコロ、行動ログ
- 自分の手番かどうかが視覚的に明確に分かること。他人の手番中は誤操作できないようにする
- PCブラウザでの利用を主対象とする。モバイル最適化は必須要件としない

## 5. 非機能要件

- **想定利用規模**：個人・友人内での利用。同時に数部屋程度が立てば十分で、大規模な同時接続設計は不要
- **通信方式**：WebSocketによるサーバー⇄クライアントのリアルタイム双方向通信
- **永続化**：MVPでは不要（サーバー再起動でゲーム状態が消えてよい）。ただし将来のログ・研究機能のため、ゲームエンジンの入出力は「イベントの列」として設計し、後から記録層を差し込みやすくしておく
- **不正防止**：サーバーがルール・乱数・勝敗判定のすべての権威を持つ（クライアントは表示と操作要求のみ）
- **実行環境**：個人開発規模。ローカル起動、または小規模PaaS（Render / Fly.io 等）へのデプロイを想定し、大掛かりなインフラは組まない

## 6. システム構成（技術スタック提案）

保守しやすさと個人開発のしやすさを優先し、以下を提案する。

| 領域 | 提案 | 理由 |
|---|---|---|
| 言語 | TypeScript統一 | フロント・バック・共有ロジックで型を共有し、カード効果やゲーム状態の型のズレを防ぐ |
| バックエンド | Node.js + Socket.IO | 「部屋（ルーム）」がSocket.IOのroom機能とそのまま対応し、実装コストが低い |
| フロントエンド | React + Vite + TypeScript | 個人開発でも立ち上げが速く、状態表示中心のUIと相性が良い |
| ゲームロジック | 共有パッケージ（`/shared`）に純粋関数のゲームエンジンを実装し、サーバーがそれを実行、クライアントは結果を表示のみ | ルールをサーバー・クライアントで二重実装しない。将来の自動対戦研究用途にもこのエンジンをそのまま再利用できる |
| 構成 | npm workspaces によるモノレポ（`/server`, `/client`, `/shared`） | 型共有と依存管理をシンプルに保つ |
| 永続化 | なし（インメモリ） | MVPスコープでは不要。将来必要になれば `/shared` のイベントログをSQLite等に書き出す層を追加する |

## 7. ゲームロジックの実装方針

- ゲーム状態を単一の `GameState` オブジェクトとして表現し、`applyAction(state, action) -> state` という純粋関数（reducerパターン）でのみ状態を変更する
- `action` はフェイズ2のA〜Gに対応する型付きコマンド（カード獲得・設置・使用・レジェンド設置・場入替・パワー獲得・変換）として定義する
- サイコロの乱数はサーバーのアクション処理時にのみ生成し、`GameState` に結果を記録する（クライアントは関与しない）
- 不正・不能なアクションは `applyAction` がエラーを返し、状態を変更しない
- この設計により、将来「AIが `applyAction` を呼ぶ」実装（自己対戦シミュレーション）や、対局ログの記録・再生（リプレイ）を後付けしやすくする

## 8. 未確定事項（今後決めること）

- ゲーム中の切断・再接続の扱い（再接続を許可するか、他プレイヤーの手番進行をどう扱うか）
- 部屋の有効期限・同時に開ける部屋数の上限
- ホストの権限範囲（強制退室（キック）機能の要否など）
- ニックネームの重複や不適切文字列の扱い
- デプロイ先（自宅サーバー／PaaS／その他）の最終決定
- AIボット対戦・拡張カード対応・リプレイ機能の着手時期（将来ロードマップ）

## 9. 開発の進め方（マイルストーン案）

1. `/shared` にゲーム状態の型とルールエンジン（本体カードのみ）を実装し、単体テストでルール（FAQ裁定含む）を担保する
2. サーバー側にSocket.IOの部屋機能を実装し、ゲームエンジンをつなぎこむ
3. 最小限のフロントエンド（ロビー＋盤面表示＋手番操作）を実装し、実際に対戦できる状態にする
4. 実プレイでの動作確認・ルール漏れの修正
5. （以降）UI磨き込み、拡張カード対応、AIボット、リプレイ・統計機能などを検討
